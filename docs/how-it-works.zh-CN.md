# 工作原理

*[English](how-it-works.md) | 中文*

机器人要把一份水或肥料从供给站送到你选的植物那里。我把它拆成移动机器人里常见的两块：一个负责**规划**走哪条路，一个负责**控制**把这条路开完。把这两件事分开，是我真正想明白它的关键，所以代码也是照这个结构写的。整个包分成 `graph.py`、`town.py`、`planner.py`、`controller.py`、`simulate.py` 和 `render.py`。

## 1. 把花园看成一张图

整座花园就是一张带权图（`graph.py`、`town.py`）。每个路口（还有每株有名字的植物，以及供给站）是一个**节点**，带一个像素坐标；每段小径是连接两个节点的**边**。边的权重就是它的长度，用两个端点和 `math.hypot` 算出来。花园小径是双向的，所以这是一张无向图：每条边都按相同权重正反各存一份。

花园就这么多。一旦小径变成了图，找一条好路线就成了标准的最短路问题，而不是这座花园特有的麻烦事。

## 2. 规划路线（Dijkstra）

给定起点（供给站）和你选的终点，规划器（`planner.py`）跑 **Dijkstra 算法** 找最短路线。

它给每个节点记一个“目前已知的最短距离”：供给站是 0，其余都是无穷大。用一个二叉堆（`heapq`）取出还没处理完的节点里最近的那个，标记为完成，再看看经过它去它的每个邻居是不是更近（这一步叫“松弛”），把变短的邻居重新压回堆里。堆里那些已经被更短路径取代的旧条目，用一个 `settled` 标记跳过就行。因为它总是先处理完最近的节点，所以终点第一次被处理完时，那个距离就是最终答案。从终点沿着 `prev` 指针往回走，就得到真正要经过的节点序列。

用堆让“取出最近的未完成节点”这一步保持在每次弹出 O(log n)，这是 Dijkstra 的教科书写法。十来个节点时直接线性扫一遍其实一样快，但堆的写法在花园变大时也撑得住。

测试里我拿一个独立的 **Floyd-Warshall** 全源最短路实现来对账，既在这座花园上对，也在几百张随机图上对，所以路线算错会被抓出来。

## 3. 沿路线行驶

规划出来的是一串节点，控制器（`controller.py`）一个一个开过去。

对当前要去的那个路口，它先算出**朝向误差**：机器人朝向和“指向该路口”的方向之间的夹角。然后按这个误差给两个轮子的速度加偏置，一个轮子比另一个快一点，机器人就转过去对准目标。误差越大转得越急。它还会往前多看一个路口：如果前面要急转弯，就在接近时提前松油门，这样能干净地过弯而不是冲过头。这个“拐弯前先减速”就是它在高速下还不偏出小径的原因。转向本身是普通的比例控制。

两个轮速通过**差速驱动**模型变成运动：前进速度是两轮的平均，转向角速度是两轮之差除以轮距。花园的 y 轴朝下（图像坐标的习惯），所以转向符号是配着这个定的（左轮快一点，机器人就往右边拐），而且有一个测试用机器人在世界坐标里的实际位移把这件事钉死，不只是看个正负号。

当机器人开到它瞄准的那个路口附近，就换成瞄准计划里的下一个。所以路口上的“决策”很简单：拐上通往规划器选中的下一个节点的那条小径。全局规划负责选弯，局部控制负责把弯拐到位。

## 4. 养护循环

一个小状态机（`simulate.py`）把这些串起来。机器人一开始在供给站待命；你选一株植物，它就取上一份水或肥料、规划“供给站→植物”、开过去；到了就放下这份水或肥料、完成一次灌溉或一次施肥，再规划“植物→供给站”、空车开回来，回到待命状态，等下一次养护。机器人身上画的那个方块表示它有没有在载货。

## 5. 画出来

渲染器（`render.py`）用 `Pillow` 把花园、规划出的虚线路线、机器人和它携带的那份水或肥料画出来，每一步画一帧，再导出成一张动图。同一份代码也生成那些静态花园地图。不依赖任何外部动画库：`Pillow` 本身就能直接保存多帧 GIF。

## 6. 浏览器播放器

同一套引擎还驱动了一个小小的浏览器演示，关键在于这里的分工。`export.py` 跑的是真正的引擎——它驱动养护状态机（`simulate.py`），由它用 Dijkstra 规划每一段、用控制器把它开完——再把输出记录成 `web/data/` 下的 JSON：花园的图（带英文名和坐标的节点、每块地接收的是水还是肥料，以及边），还有每株植物的两段规划节点路径，加上机器人每一个仿真步的位姿（`x`、`y`、`theta`、`carrying`）。用下面这条命令重新生成：

```bash
python -m delivery_robot.export web/data
```

`web/` 里的网页（`index.html`、`player.js`、`style.css`）**只负责播放**。它加载这些 JSON，在 `<canvas>` 上画出来：花园小径、规划出的虚线路线，以及一帧帧移动的机器人，带植物选择、播放/暂停和速度滑块。它里面没有任何算法——JavaScript 里没有 Dijkstra、没有转向、没有运动学。两个记录帧之间，它会对位置和角度做线性插值，让播放看起来顺滑，但那是为了画图的插值，不是仿真。速度滑块只改变播放速度，绝不会重新驱动引擎。所以实质性的代码都留在有测试覆盖的 Python 里；浏览器只是引擎已算好结果的一个查看器。界面按钮可以在 English 和中文之间切换，但地图上画出来的植物名保持英文。

## 7. 它是什么，不是什么

这是我为了研究这个思路写的一个软件模型。花园是抽象的，距离单位是像素，不是米。规划器和控制器都是教科书里的标准做法，我把每一步都摊开写，好让自己跟得下来。它不是任何实体机器人上的固件，也没有细致地建模传感器、电机或时序。我的目的就是搞懂路线规划和循迹是怎么配合起来的，并且能测出它们确实管用（见 [`../tests/run.py`](../tests/run.py)）。
